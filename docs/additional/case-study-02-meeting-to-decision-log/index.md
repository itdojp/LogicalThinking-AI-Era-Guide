# ケーススタディ2：会議アジェンダ→議事録→意思決定ログ

## 1. 目的 / 読み手 / 制約 / 機密区分

- 目的: 会議の「決定事項」と「次アクション」を確実に残し、関係者の認識差を減らす
- 読み手: 参加者＋関係者（欠席者含む）
- 制約: 発言のニュアンスは人が最終確認する。不明な点は推測しない
- 機密区分: 個人評価・顧客固有情報は入力しない（抽象化する）

標準手順は [AI活用の標準業務フロー（1枚）](../../introduction/02-standard-workflow/) を参照してください。

## 2. 初期プロンプト（例）

```text
次回の定例会議のアジェンダを作ってください。会議後は議事録も作りたいです。
```

## 3. 初回出力（抜粋）と課題（例）

```text
アジェンダ:
1. 進捗共有
2. 課題共有
3. 次のアクション
議事録は会議後に作成します。
```

主な課題:
- 会議の目的・意思決定点が明示されていない（「共有」で終わる構成）
- 議事録の形式（決定/ToDo/未決）が定義されていない
- 会議後の「意思決定ログ」まで接続していない

## 4. CARE評価（問題点と改善方針）

- 正確性: 会議の前提情報が不足 → 目的/参加者/意思決定点/制約を入力する
- 適切性: 運用に必要な粒度（担当・期限）がない → 出力フォーマットを固定する
- 関連性: 重要論点が抜ける → 事前資料や懸念点を入力する
- 効果性: 会議後の行動に繋がらない → 決定事項/ToDo/未決/判断理由をログ化する

## 5. 改善プロンプト（例）

```text
【背景】部門横断の週次定例です。今回の目的は「次週の優先順位と担当」を決めることです。
【参加者】開発/運用/営業の各リーダー（計6名）
【意思決定点】次の2点を会議内で確定したい:
  1) 来週の優先度上位3件（理由付き）
  2) 各タスクの担当と期限（暫定でも可）
【制約】未確定は未確定のまま書く。推測で補完しない。
【成果物】次の2つを出力:
  A) アジェンダ案（各項目に目的/意思決定点/時間配分/事前資料）
  B) 議事録テンプレ（決定事項/ToDo/未決事項/判断理由/次回議題）
```

## 6. 改善後出力（抜粋）

```text
A) アジェンダ案（60分）
1. 目的と意思決定点の確認（5分）
2. 重要課題の共有（15分）
3. 優先度上位3件の確定（25分）
4. 担当・期限の確定（10分）
5. 次回議題と未決の整理（5分）

B) 議事録テンプレ
- 決定事項:
- ToDo（担当/期限）:
- 未決事項（保留理由/次の確認事項）:
- 判断理由（根拠/前提）:
- 次回議題:
```

## 7. 最終成果物（意思決定ログのサンプル）

会議後は、議事録に加えて「意思決定ログ」を残すと、後からの再現性が上がります（内容はサンプル）。

```text
# 意思決定ログ（週次定例 / YYYY-MM-DD）

## 決定事項
- 来週の優先度上位3件:
  1) 障害対応の恒久対策（理由: 再発防止が最優先）
  2) 重要顧客の期限対応（理由: 期限が固定で影響が大きい）
  3) 運用手順の見直し（理由: 作業コストの削減が見込める）

## ToDo（担当/期限）
- 恒久対策案の作成: 開発A / 金曜まで
- 顧客向け調整案の作成: 営業B / 水曜まで
- 手順見直しの現状棚卸し: 運用C / 来週火曜まで

## 未決事項（保留理由/次の確認事項）
- 追加要員の投入可否: 予算承認が必要（次回までに上長確認）

## 判断理由（根拠/前提）
- 優先度は「影響度×緊急度」を基準にした
- 顧客対応は契約期限が前提（要確認: 期限変更の余地）
```

## 8. 承認ゲート / 責任者 / ログ（最低限）

- 責任者（最終）: 会議体のオーナー（例: 部門長/プロジェクト責任者）
- 承認ゲート（例）: 会議内で暫定決定 → 関係者レビュー → 次回で確定
- ログ（最低限）:
  - 会議目的と意思決定点（毎回固定）
  - 議事録（決定/ToDo/未決）
  - 意思決定ログ（判断理由と前提）
