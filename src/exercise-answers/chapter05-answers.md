# 第5章 演習問題 解答・解説

## 演習5-1：指示設計シートを作る

### 解答例

```text
【指示設計シート】
目的:
次回提案で、当社の導入後支援を主軸にするか判断するため、競合比較の初稿を作る。

読み手 / 承認者:
営業部長、提案責任者、必要に応じて法務・情報管理担当。

意思決定点:
次回提案で、価格訴求、導入後支援、移行支援のうち、どれを主軸にするか。

使ってよい情報:
競合の公開Webページ、公開資料、当社の公開サービス説明、匿名化済みの顧客課題メモ。

使ってはいけない情報:
顧客名、個別契約条件、未公開価格、社内限定の提案資料、個人情報。

前提 / 用語:
「導入後支援」は初期設定、問い合わせ対応、運用定着支援を含む。価格情報は公開情報のみ扱う。

AIに依頼する作業:
公開情報を基に、競合比較表の初稿を作る。推定と公開情報を分け、提案への示唆を列挙する。

人間が判断する作業:
公開情報の正確性、競合比較の妥当性、提案で使える表現、法務・知財・情報管理上の問題有無。

出力形式:
競合名 / 公開情報の要約 / 根拠URLまたは資料名 / 要確認事項 / 提案への示唆 / 使ってよい表現。

必須項目:
根拠候補、要確認事項、推定ラベル、社外提案で使う前の確認点。

受け入れ基準:
公開情報と推定が分かれている。根拠のない数値断定がない。要確認事項が明示されている。機密情報を含まない。

差し戻し条件:
未確認情報を断定している。競合名や公開資料の出所が不明。顧客名や契約条件を含む。提案主軸の判断に関係しない情報が多い。

ログ化 / 再利用条件:
確認日、利用した公開資料、修正履歴、社外提案への転用可否を記録する。競合情報は再利用前に再確認する。
```

### 解説

「調べて報告して」だけでは、AIは調査範囲や利用可否を判断できません。解答例では、目的、意思決定点、情報分類、出力形式、受け入れ基準を先に決め、AI出力を評価しやすくしています。

## 演習5-2：context pack を作る

### 解答例

```text
【context pack】
目的:
会議メモを、関係者が次アクションを確認できる意思決定ログに変換する。

読み手:
会議参加者、プロジェクト責任者、次回会議のファシリテーター。

意思決定点:
決定済み事項、未決事項、担当者、期限、次回確認事項を共有してよいか。

利用資料:
- 会議メモ: 社内限定。個人名は役割名へ置換する。
- アジェンダ: 社内限定。議題の順序確認に使う。
- 関連タスク一覧: 社内限定。担当と期限の確認に使う。

用語:
「決定事項」は、会議内で合意した内容のみを指す。「ToDo」は担当者と期限があるものに限定する。

前提:
会議メモは走り書きであり、発言者名や日付に誤記がある可能性がある。

除外する情報:
雑談、個人評価、未承認の人事情報、顧客の実名、契約前提。

匿名化ルール:
個人名は「担当者A」「部門長B」に置換する。顧客名は「顧客X」に置換する。

AIに質問してよいこと:
未決事項の候補、ToDoの抜け漏れ、矛盾している箇所、要確認事項。

AIに判断させないこと:
正式な決定の確定、担当者の変更、期限の確定、承認済みとしての扱い。
```

### 解説

context pack では、AIに渡す資料を列挙するだけでなく、使う範囲、除外情報、匿名化ルールを明示します。これにより、会議メモに含まれる機微情報をそのまま出力へ混入させるリスクを下げられます。

## 演習5-3：output schema と acceptance criteria を作る

### 解答例

#### output schema

```text
【経営向け1枚サマリー schema】
1. 判断点
2. 結論候補
3. 背景
4. 根拠候補
5. 選択肢
6. 推奨案
7. 想定反論
8. 主なリスク
9. 採用条件
10. 保留条件
11. 要確認事項
12. 次アクション
```

#### 受け入れ基準

| 項目 | 基準 |
| --- | --- |
| 判断可能性 | 読み手が採用、保留、差し戻しの判断を選べる |
| 根拠 | 主張ごとに根拠候補または要確認が付いている |
| 簡潔性 | 1枚で読める粒度に整理されている |
| リスク | 採用時のリスクと保留条件が明記されている |
| 情報分類 | 機密情報、個人情報、未承認情報を含まない |
| 承認 | 承認者と確認事項が分かる |

#### 差し戻し条件

- 判断点がない。
- 根拠のない数値や外部情報を断定している。
- 社外共有不可の情報を含んでいる。
- 推奨案だけで、代替案、反論、保留条件がない。
- 読み手が次に何をすべきか分からない。

### 解説

schema は出力の見た目だけではなく、評価と承認のための構造です。根拠、要確認、リスク、採用条件を入れることで、第6章の評価や第4章の採否判断につなげやすくなります。

## 演習5-4：段階的指示を設計する

### 解答例

```text
【第1段階: 理解確認】
以下の営業ヒアリングメモを読み、顧客課題、顧客の意思決定点、制約、要確認事項を要約してください。顧客名と個人名は匿名化してください。不明な点は推測せず、質問として列挙してください。

【第2段階: 構成案】
理解確認に基づき、提案骨子の構成案を3案作ってください。各案について、向いている条件、弱点、必要な追加確認を示してください。

【第3段階: 初稿】
最も実現可能性が高い案を基に、提案骨子の初稿を作ってください。出力形式は「判断点 / 顧客課題 / 提案方針 / 根拠候補 / 想定反論 / リスク / 要確認事項 / 次アクション」としてください。

【第4段階: 自己点検】
初稿を次の受け入れ基準で点検してください。顧客課題と提案方針が対応していること、根拠候補と推定が分かれていること、機密情報を含まないこと、次アクションが明確であること。

【第5段階: 差し戻し】
点検結果で不足がある箇所を修正してください。根拠がない主張は要確認へ移し、顧客名や個人名が残っている場合は匿名化してください。
```

### 解説

段階的指示では、最初から完成版を作らせません。理解確認、構成案、初稿、自己点検、差し戻しを分けることで、前提のずれや機密混入を早い段階で発見できます。

## 演習5-5：手段選択を行う

### 解答例

| 手段 | 要否 | 理由 | 承認条件 / ログ化 |
| --- | --- | --- | --- |
| 通常生成 | 必要 | 回答案の文面、表現の言い換え、要確認事項の整理に使える | 生成した文面と修正履歴を残す |
| 検索 / retrieval | 必要 | 社内FAQと公開サービスページを参照し、最新の正本情報に基づく必要がある | 参照元、版、確認日を出力に残す |
| 表やPDFの読み取り | 条件付き | 直近問い合わせ一覧が表形式またはPDFの場合に必要 | 個人情報や顧客名を匿名化してから扱う |
| tool use / function calling | 条件付き | FAQ更新候補を社内システムへ反映する場合に必要 | 書き込み操作は人間承認後。実行ログを残す |
| MCP連携 | 条件付き | 社内FAQ、問い合わせ管理、公開ページ情報を安全に接続する運用がある場合に有効 | 接続先、権限、取得範囲、監査ログを確認する |

### 判断例

まずは、社内FAQと公開サービスページを検索 / retrieval 対象にし、直近問い合わせ一覧は匿名化した表として扱います。AIには回答テンプレートの初稿、根拠候補、要確認事項を作らせます。社内FAQへの反映や公開ページ更新は、AIに直接実行させず、人間のレビューと承認後に行います。

### 解説

手段選択では、「便利だから連携する」のではなく、必要な根拠、権限、承認、ログ化を確認します。特に社内システムへの書き込みや顧客向け公開情報の更新は、人間の承認を前提にします。

## 章全体の確認ポイント

- 指示設計は、プロンプト文の作成ではなく、業務成果物の完了条件を設計する作業です。
- context pack では、利用資料、除外情報、匿名化ルール、AIに判断させないことを明示します。
- output schema と acceptance criteria を先に決めると、評価と差し戻しが容易になります。
- 段階的指示は、前提確認、構成、初稿、自己点検、差し戻しに分けます。
- tool use / function calling / MCP連携は、権限、承認、ログ化とセットで判断します。
