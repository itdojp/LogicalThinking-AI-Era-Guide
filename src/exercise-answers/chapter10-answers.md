# 第10章 演習問題 解答・解説

この解答例は、唯一の正解ではありません。実務では、自社の情報分類、AI利用ルール、承認フロー、契約条件、法務・セキュリティ部門の判断に合わせて調整してください。

## 演習10-1：document task brief を作る

### 題材: 営業提案書の初稿作成

```text
成果物名: 顧客A向け業務改善提案書 初稿
文書種別: 提案書
利用目的: 次回商談で提案範囲と検討ステップを合意する
読み手: 顧客部門長、情報システム担当、当社提案責任者
読み手の関心事: 課題理解、導入負荷、セキュリティ、費用感、次アクション
意思決定点: 詳細ヒアリングと概算見積へ進むか
依頼したい判断: 次回ワークショップ実施の合意
期限: 次回商談前日
情報分類: 顧客情報、社内限定、公開情報
AI利用環境: 社内AI。外部AIには顧客名、契約条件、見積情報を渡さない
参照してよい資料: 公開サービス資料、匿名化した課題メモ、社内提案テンプレート
AIに渡してはいけない資料: 顧客名、担当者名、個別見積、契約条件、未公開ロードマップ
確認済みの根拠: 公開サービス説明、ヒアリングで合意済みの課題要約
未確認事項: 予算上限、導入時期、既存契約制約、セキュリティ要件詳細
承認者: 提案責任者、必要に応じて法務・情報システム
最終判断者: 顧客部門長
再利用条件: 顧客固有情報を削除した構成のみ再利用可
```

### 解説

文書作成前に、読み手、意思決定点、情報分類、AI利用環境を決めています。AIには提案骨子や表現案を作らせますが、顧客固有情報、価格、契約条件、法務判断は人間が扱います。

## 演習10-2：one-page summary canvas を作る

### 題材: 部門長向けAI利用試験導入の承認依頼

```text
タイトル: 営業提案書作成における社内AI試験導入の判断メモ
読み手: 営業部門長
判断してほしいこと: 2週間の試験導入を承認するか
結論 / 推奨: 限定範囲での試験導入を推奨
背景: 提案書初稿作成で、課題整理と構成作成に時間がかかっている
主要根拠:
- 提案書は proposal skeleton に分解でき、AIに骨子案を作らせやすい
- 顧客名、価格、契約条件を除外すれば、入力リスクを管理しやすい
- 第9章の policy / approval matrix に沿って提案責任者の承認を組み込める
選択肢:
1. 限定試験を実施する
2. 全面導入する
3. 導入を保留する
推奨案: 1. 限定試験を実施する
リスク / 制約:
- 未確認の効果数値を顧客向け文書に入れない
- 顧客固有情報を扱う場合は社内AI環境に限定する
- 最終版は提案責任者が承認する
未確認事項: 試験対象案件、利用ログ保管場所、法務確認要否
依頼事項: 試験導入範囲とレビュー担当者の承認
次アクション: 対象案件を1件選定し、task brief と review checklist を作成する
```

### 解説

1枚サマリーは、背景説明を圧縮するだけではありません。判断対象、推奨案、選択肢、リスク、未確認事項、依頼事項を分けることで、読み手が判断しやすくなります。

## 演習10-3：proposal skeleton を作る

### 題材: 社内研修プログラム導入提案

```text
1. 表紙 / 提案目的
   - AI時代の論理的文書作成研修の試験導入提案
2. 相手の課題理解
   - 報告書、稟議、提案書の品質が担当者ごとにばらつく
   - AI出力の根拠確認と承認フローが属人化している
3. 現状と影響
   - 文書レビューが後工程に集中し、差し戻しが発生している
   - 未確認の数値や一般論が残るリスクがある
4. 提案方針
   - document task brief、evidence-to-document traceability matrix、document review checklist を標準化する
5. 解決策の範囲
   - 1枚サマリー、報告書、稟議メモを対象にする
   - 顧客向け提案書は第2段階で扱う
6. 導入ステップ
   - 研修設計、パイロット、レビュー、テンプレート改訂、展開判断
7. 期待できる効果の扱い
   - 効果数値はサンプルとして扱い、試験導入後に実測する
8. 前提条件 / 除外範囲
   - 法務文書や契約書の最終判断は対象外
   - 機密情報は社内承認済み環境のみで扱う
9. リスクと対策
   - AI出力の断定: 未確認事項欄を必須化
   - 個人情報混入: data classification card を必須化
10. 体制 / 役割分担
   - 研修責任者、部門レビュー担当、情報システム確認者
11. 費用 / 契約条件の扱い
   - 研修実施費、教材作成工数、利用環境の範囲を別紙で確認
12. 次アクション
   - パイロット対象部門と実施日を決める
```

### 想定反論と扱い

| 想定反論 | 回答方針 | 本文での扱い |
| --- | --- | --- |
| 研修効果が測りにくい | 文書レビュー観点の定着度で評価する | 効果測定計画へ記載 |
| 現場負荷が増える | 既存テンプレートへ組み込む | 導入ステップへ記載 |
| AI利用リスクがある | 情報分類と承認フローを前提にする | リスク対策へ記載 |

### 解説

提案書は、自社の主張だけでなく、相手の課題、制約、懸念、合意条件を扱います。期待効果の数値は、実測または根拠がない限り断定しません。

## 演習10-4：report / approval memo schema を適用する

### 題材: 月次問い合わせ増加の報告書

```text
件名: 5月問い合わせ増加に関する月次報告
報告目的: 問い合わせ増加の状況、仮説、次アクションを共有する
対象期間 / 対象範囲: 5月、サポート窓口全体
確認済み事実:
- 問い合わせ管理表上、5月の問い合わせ件数は前月より増加した
- 新機能に関する問い合わせが目立つ
観察された変化:
- 操作手順、権限設定、エラー表示に関する質問が増えている
解釈 / 仮説:
- 新機能公開後のオンボーディング情報が不足している可能性がある
- ヘルプページの導線が分かりにくい可能性がある
根拠:
- 問い合わせ管理表
- FAQ閲覧ログ
- サポート担当者の分類メモ
要確認事項:
- 増加が一時的か継続的か
- FAQ閲覧ログと問い合わせ分類が同じ粒度で比較できるか
リスク:
- 根本原因を誤ると、FAQ更新だけでは改善しない
推奨アクション:
- 上位問い合わせ3分類を確認し、FAQと画面内説明の改善案を作る
担当者 / 期限:
- サポート責任者が次回定例までに改善案を提示する
```

### 稟議メモへ変換する場合

```text
承認してほしいこと: FAQと画面内説明を改善するための作業枠を確保する
推奨案: 上位問い合わせ3分類に限定して改善する
代替案: 全FAQを見直す / 問い合わせ対応を増員する / 保留する
リスクと対策: 原因仮説が外れる可能性があるため、翌月に再評価する
却下または保留時の影響: 問い合わせ対応負荷が継続する可能性がある
承認条件: 対象範囲、担当者、確認指標、次回レビュー日を決める
```

### 解説

報告書では、事実、解釈、仮説、推奨を分けます。稟議へ変換する場合は、承認してほしいこと、代替案、リスク、承認条件を追加します。

## 演習10-5：evidence-to-document traceability matrix と document review checklist を作る

### evidence-to-document traceability matrix例

| 文書内の主張 | 根拠 | 出所 | 確認日 | 状態 | 本文での扱い |
| --- | --- | --- | --- | --- | --- |
| 問い合わせ件数が増加した | 問い合わせ管理表 | サポート管理ツール | 作成時に記録 | 確認済み | 本文へ採用 |
| 新機能関連の問い合わせが目立つ | 分類メモ | サポート担当者分類 | 作成時に記録 | 条件付き確認 | 「目立つ」と表現し、件数は別途確認 |
| FAQ導線が原因である | 仮説のみ | なし | 未確認 | 要確認 | 仮説として扱う |
| FAQ改善で問い合わせが大幅減少する | 根拠なし | なし | なし | 未採用 | 本文から削除 |

### document review checklist例

```text
目的:
☑ 報告目的と読み手が明確である
☑ 読み手が判断することが分かる
☑ 今回決めないことが分かる

論理:
☑ 事実、解釈、仮説、推奨が分かれている
☑ 推奨アクションと根拠がつながっている
☑ 代替案または保留条件を示している

根拠:
☑ 出所と確認日を記録した
☑ 未確認の効果数値を本文で断定していない
☑ 根拠なしの主張を削除した

リスク:
☑ 情報分類を確認した
☑ 個人名や顧客固有情報を本文から除外した
☑ 誤った原因推定のリスクを記載した

承認:
☑ 担当者と期限を明記した
☑ 次回レビュー条件を残した
☑ 再利用時に必要なテンプレート更新点を記録した
```

### 解説

evidence-to-document traceability matrix は、AIが作った整った文章を根拠確認できる文書へ変えるために使います。document review checklist は、文章表現だけでなく、目的、論理、根拠、リスク、承認を点検します。

## まとめ

- document task brief は、文書作成前に目的、読み手、意思決定点、情報分類を定義するために使います。
- one-page summary canvas は、結論、根拠、リスク、未確認事項、依頼事項を1枚に集約します。
- proposal skeleton は、相手の課題、提案範囲、反論、合意条件を整理します。
- report / approval memo schema は、報告と承認依頼を混同しないために使います。
- evidence-to-document traceability matrix は、主張、根拠、出所、確認日、未確認事項をつなぎます。
- document review checklist は、CARE+、読み手適合、根拠、リスク、承認、再利用条件を確認するレビュー資産です。
