# 第4章 演習問題 解答・解説

## 演習4-1：問題設定メモを作る

### 解答例

```text
【問題設定メモ】
目的:
社内研修の参加率改善について、AI活用を含む実行可能な改善案を比較し、次回四半期で試す施策を決める。

読み手 / 承認者:
人事部長、各部門長、研修運営責任者。

現象:
任意参加の社内研修で、申し込み後の欠席や未受講が目立つ。具体的な参加率、欠席率、部門別差は要確認。

影響:
研修で扱う業務ルールやAI利用ポリシーが浸透せず、部門ごとの運用差、問い合わせ増加、リスク理解不足につながる可能性がある。

解くべき問題:
参加率の低さそのものではなく、対象者が必要な研修を期限内に受講し、業務で使える状態になっていないこと。

今回の意思決定点:
次回四半期に、通知方法、上長承認、研修形式、フォローアップ、AIによる個別リマインドのどれを優先して試すか。

対象範囲:
全社必須ではなく、まずAI利用頻度が高い部門向けの研修を対象にする。

対象外:
人事評価制度の変更、懲戒、外部研修会社の選定。

制約:
個人の受講履歴は社内規程に従って扱う。個人情報を外部AIへ入力しない。部門長への共有範囲は人事ポリシーに従う。

利用できる根拠:
研修申し込み数、受講完了数、欠席理由アンケート、部門別参加傾向、受講後アンケート、問い合わせ件数。

未確認事項:
必須研修か任意研修か、参加率の定義、部門長が確認できる情報範囲、既存LMSの通知機能。

採否判断に使う評価軸:
参加率改善への寄与、受講者負担、部門長の運用負荷、個人情報保護、実施コスト、再利用性。
```

### 解説

「参加率が低い」という現象だけでは、改善案の採否判断ができません。解答例では、目的、読み手、意思決定点、対象範囲、制約を先に置き、AIに渡してよい情報と人間が確認すべき情報を分けています。

## 演習4-2：仮説検証シートを作る

### 解答例

```text
【仮説検証シート】
問題:
対象者が必要な研修を期限内に受講し、業務で使える状態になっていない。

仮説:
もし欠席や未受講の主要因が「受講の優先度が分からないこと」と「上長からの期待が明確でないこと」であれば、部門長向けの確認テンプレートと受講者向けの個別リマインドを整えることで、期限内受講率が改善するはずである。

根拠候補:
過去アンケートの自由記述、欠席理由、部門別受講率の差、リマインド送信後の受講率変化。

反証される条件:
未受講理由の大半が日程不一致、業務繁忙、研修内容の不適合であり、優先度認識や上長期待と関係が薄い場合。

確認するデータ / 資料:
受講完了ログ、欠席理由アンケート、研修日程、部門別業務繁忙期、既存通知文面、部門長への案内履歴。

AIに依頼する作業:
欠席理由アンケートの匿名化済み集計から原因候補を分類する。受講者向けリマインド文と部門長向け確認テンプレートの初稿を作る。反論とリスクを列挙する。

人間が確認する作業:
個人情報の扱い、部門長に共有してよい範囲、研修の必須性、通知文のトーン、現場負荷。

小さく試す方法:
2部門を対象に、匿名化した受講状況の週次確認と、部門長からの期待明示テンプレートを1か月試す。

採用条件:
個人情報保護上の問題がなく、部門長の追加負荷が許容範囲で、対象部門の期限内受講率に改善傾向が見られる。

保留 / 中止条件:
受講者の心理的負担が大きい、ランキング公開など過度な圧力になる、部門長の運用負荷が高すぎる、参加率以外の学習効果が低下する。

次アクション:
匿名化済みデータで原因仮説を確認し、パイロット案を人事部長と対象部門長へ確認する。
```

### 解説

仮説は「AIでリマインドすれば参加率が上がる」と短絡させません。原因、施策、観測する変化、反証条件を分けることで、検証可能になります。

## 演習4-3：AI出力の評価rubricを作る

### 解答例

#### 必須条件

| 必須条件 | 判定基準 |
| --- | --- |
| 個人情報保護 | 個人の受講履歴、評価、健康情報などを外部AIへ入力しない設計になっている |
| 目的整合 | 参加率だけでなく、必要な研修を業務で使える状態にする目的に合っている |
| 根拠欄 | 施策ごとに根拠候補、確認済み事項、未確認事項が分かれている |
| 過度な圧力の回避 | ランキング公開、懲罰的通知など、心理的安全性を損なう案をそのまま採用しない |
| 承認条件 | 人事、部門長、必要に応じて法務・情報管理の確認点が明記されている |

#### 加点評価

| 評価軸 | 3: 採用水準 | 2: 条件付き | 1: 差し戻し |
| --- | --- | --- | --- |
| 判断可能性 | 施策、対象、期限、判断点が明確 | 一部の対象や期限が曖昧 | 何を決めるか不明 |
| 実現可能性 | 既存の運用やツールで小さく試せる | 一部追加準備が必要 | 大規模変更が前提 |
| 受講者負担 | 負担を増やさず、受講しやすさを高める | 負担増への対策が必要 | 受講者への圧力が強い |
| 効果測定 | 受講率、理解度、業務適用の指標がある | 参加率のみ測る | 測定方法がない |
| 再利用性 | 他研修にも使えるテンプレートがある | 一部修正で再利用可能 | 個別対応に依存する |

### 解説

必須条件と加点評価を分けることが重要です。個人情報保護や過度な圧力の回避は、総合点で補える項目ではありません。必須条件を満たさない案は、条件付き採用ではなく差し戻しまたは保留にします。

## 演習4-4：採否判断を行う

### 解答例

**分類: 差し戻し**

### 判断理由

- 「全社員に義務化」が、今回の対象範囲や研修の必須性を確認せずに断定されています。
- 未参加者を部門長へ通知する範囲、個人情報保護、社内ポリシー、心理的安全性が確認されていません。
- ポイント付与やランキング公開は、過度な競争や不適切な個人情報の露出につながる可能性があります。
- 参加率を上げる手段に偏っており、研修内容の妥当性、日程、受講者負担、業務適用度を確認していません。
- 根拠候補、採用条件、保留条件が示されていません。

### 次アクション

1. 研修が必須か任意か、対象者、部門長へ共有してよい情報範囲を確認します。
2. 個人名やランキング公開を前提にしない改善案へ差し戻します。
3. 受講者負担、研修内容、日程、上長からの期待明示、リマインド方法を比較するよう再依頼します。
4. 施策ごとに、根拠候補、未確認事項、リスク、承認者を明記させます。

### 差し戻し指示例

```text
次の理由で差し戻します。

1. 個人情報保護: 未参加者の個人情報を部門長へ通知する範囲が未確認です。
2. 受講者負担: 義務化、ポイント、ランキング公開は心理的負担や不公平感のリスクがあります。
3. 根拠: 参加率低下の原因が未確認のまま施策を断定しています。
4. 目的整合: 参加率だけでなく、研修内容を業務で使える状態にする観点が不足しています。

個人名の公開やランキングを前提にせず、次の形式で再提案してください。
- 原因仮説
- 必要な確認データ
- 施策案
- 受講者負担
- 個人情報保護上の注意
- 採用条件
- 保留条件
```

### 解説

このAI出力案は、効果がありそうな施策を強く断定していますが、情報分類、社内ポリシー、受講者負担、根拠確認が不足しています。実務では、すぐに採用せず、評価軸と受け入れ基準に沿って差し戻します。

## 演習4-5：decision memo にまとめる

### 解答例

```text
【AI出力採否判断メモ】
対象成果物:
社内研修の参加率改善案の初稿。

依頼したタスク:
社内研修の参加率を改善する施策案の作成。

採否区分:
差し戻し。

判断理由:
個人情報保護、受講者負担、社内ポリシー、根拠確認が不足している。義務化、部門長通知、ランキング公開を断定しており、対象範囲や研修の必須性を確認していない。

満たした受け入れ基準:
参加率を上げる施策候補は提示されている。

満たしていない受け入れ基準:
個人情報保護、過度な圧力の回避、根拠欄、承認条件、目的整合。

根拠確認の状況:
未確認。受講完了ログ、欠席理由、研修内容への満足度、部門別傾向、既存通知文面を確認する必要がある。

主なリスク:
個人情報の不適切共有、受講者への過度な圧力、形式的参加の増加、部門長の運用負荷増加、研修内容そのものの問題を見落とすこと。

人間が修正する点:
対象範囲、共有可能な情報、承認者、リスク許容度を明確にする。ランキング公開を削除し、匿名化集計や任意のフォローアップを中心に再設計する。

承認者 / 確認日:
人事部長、情報管理責任者、対象部門長。確認日は実施時に記録する。

再利用時の注意:
研修の必須性、対象者、個人情報の共有範囲が異なる場合は、そのまま再利用しない。必ず問題設定メモと評価rubricを更新する。
```

### 解説

decision memo では、採否区分だけでなく、満たした条件、満たしていない条件、根拠確認、リスク、承認者を残します。これにより、AI出力を使わなかった理由も、次の改善指示も明確になります。

## 章全体の確認ポイント

- 問題設定は、AIへの依頼文の前段ではなく、採否判断の基準です。
- 仮説は、根拠候補、反証条件、確認データとセットで扱います。
- 評価rubricでは、必須条件と加点評価を分けます。
- AI出力は、採用、条件付き採用、差し戻し、保留の4分類で次アクションに接続します。
- 採否判断は、承認、監査、再利用のために短くてもログとして残します。
