# 第16章 演習問題 解答・解説

この解答例は、唯一の正解ではありません。実務では、自社の情報分類、AI利用ルール、承認フロー、利用可能なツール、顧客契約、法務・セキュリティ・情報システムの所管部門判断に合わせて調整してください。

## 演習16-1：project outcome brief を作る

### 題材: 営業提案資料作成プロセスへのAI導入

```text
プロジェクト名: 営業提案資料作成プロセスへのAI導入
背景:
- 初回提案資料の構成、根拠確認、FAQ作成が担当者ごとにばらついている
- 顧客共有前レビューで、根拠不足、未確認事項、承認者不明による差し戻しが発生している
解決したい業務課題:
- 提案資料の入口設計、根拠確認、レビュー基準を標準化する
意思決定点:
- 試行後に、対象チームを広げるか、テンプレートを修正するか、中止するかを判断する
主な読み手 / 利用者:
- 営業担当、営業責任者、営業企画、法務、情報システム
対象成果物:
- proposal skeleton、customer FAQ draft、evidence matrix、review workflow board
対象外:
- 価格条件、契約条件、正式回答、法的見解、顧客固有情報を含む外部AI入力
成功条件:
- 顧客共有前レビューで、根拠、未確認事項、承認者が確認できる状態になる
受け入れ基準:
- 提案資料ごとに task brief、根拠表、レビュー状態、承認者が記録されている
扱う情報分類:
- 社内限定、顧客情報、契約情報、価格情報
利用するAI環境 / ツール:
- 社内AIのみ。顧客名、価格、契約条件は入力しない
主要ステークホルダー:
- 営業担当、営業責任者、営業企画、法務、情報システム、顧客責任者
承認者:
- 試行承認は営業企画マネージャー、本格展開は営業責任者
主要リスク:
- 情報分類漏れ、未確認効果の断定、承認前の対外共有
未確認事項:
- セキュリティ標準文の最新版、価格表共有可否の承認条件
ログ化先:
- review workflow board、change request decision log、post-implementation learning report
次回判断日:
- 試行完了後のレビュー会
```

### 解説

「AIを導入する」ではなく、どの成果物のどの品質を改善するかを明示しています。対象外と承認者を含めることで、便利さだけで進めない計画になっています。

## 演習16-2：KPI / leading indicator map を作る

### KPI / leading indicator map例

| 区分 | 指標 | データ出所 | 判断に使う場面 | 注意点 |
| --- | --- | --- | --- | --- |
| 最終成果 | 顧客共有前レビューで必要項目がそろっている割合 | review workflow board | 本格展開可否 | 割合だけでなく不足理由も見る |
| 先行指標 | task brief と evidence matrix の作成有無 | 提案資料フォルダ | 試行中の運用確認 | 作成しただけで品質を判断しない |
| 品質指標 | 差し戻し理由の分類 | レビューコメント | テンプレート改善 | 件数よりも原因を見る |
| リスク指標 | 情報分類に関する指摘 | レビュー記録 | 一時停止判断 | 重大リスクは件数より内容を重視する |
| 運用負荷 | 承認待ちの滞留理由 | review workflow board | 承認フロー見直し | 速さだけを追わない |
| 再利用 | 利用条件付きで再利用されたテンプレート | prompt / template asset register | 資産保守 | 古い前提の惰性利用に注意する |

### 解説

利用回数や作成件数だけでなく、品質、リスク、運用負荷、再利用を見ています。指標は個人評価ではなく、運用改善のために使います。

## 演習16-3：AI adoption roadmap と pilot evaluation plan を作る

### AI adoption roadmap例

```text
導入テーマ: 営業提案資料作成プロセスへのAI導入
対象業務: 初回提案資料、FAQ、根拠表、レビュー依頼
対象チーム: 法人営業提案チーム
Phase 1 準備:
- 現行提案資料の差し戻し理由を分類する
- 情報分類、対象外、承認者を整理する
- proposal skeleton と evidence matrix を用意する
Phase 2 試行:
- 限定チームで社内AIのみを使う
- 顧客名、価格、契約条件は入力しない
- review workflow board にレビュー状態を残す
Phase 3 評価:
- pilot evaluation plan に沿って、品質、リスク、運用負荷を確認する
- 差し戻し理由と承認滞留を分析する
Phase 4 展開:
- テンプレート、FAQ、教育資料を更新して対象チームを広げる
- 法務・情報システムへの相談条件を明示する
Phase 5 定着:
- 月次で指標とテンプレートを見直す
- 古い前提のテンプレートを廃止する
各Phaseの判断条件:
- 情報分類の重大指摘がないこと
- 承認者と相談先が明確であること
- テンプレート改善点が記録されていること
責任者: 営業企画マネージャー
関係者: 営業責任者、法務、情報システム
ログ化先: post-implementation learning report
```

### pilot evaluation plan例

```text
試行名: 初回提案資料AI支援パイロット
対象業務: 提案骨子、FAQ案、根拠表の作成
対象者 / 対象チーム: 法人営業提案チームの限定メンバー
試行期間: 営業レビュー会まで
使うテンプレート / 資産: proposal skeleton、customer FAQ draft、evidence matrix、review workflow board
扱う情報分類: 社内限定、顧客情報、価格情報。顧客固有情報は入力しない。
評価したい仮説:
- proposal skeleton と evidence matrix を使うと、レビュー前に論点抜けと根拠不足を発見しやすくなる
受け入れ基準:
- 成果物ごとに目的、読み手、根拠、未確認事項、承認者が記録されている
測定する指標:
- 差し戻し理由、情報分類指摘、承認滞留、再利用できたテンプレート
確認するリスク:
- 情報分類漏れ、未確認効果の断定、承認前の対外共有
除外する業務:
- 価格条件、契約条件、正式回答、法的見解
レビュー方法:
- 営業責任者が一次レビューし、必要に応じて法務・情報システムへ相談する
継続条件:
- 重大な情報分類指摘がなく、テンプレート改善点が明確である
修正条件:
- 差し戻しが特定項目に集中する
中止条件:
- 入力禁止情報の扱いに重大な問題が見つかる
本格展開前の承認者:
- 営業責任者
```

### 解説

ロードマップでは段階を分け、パイロットでは検証する仮説と中止条件を定義しています。これにより、「使ってみた」だけで終わらない試行になります。

## 演習16-4：change impact / stakeholder map と risk / dependency / assumption register を作る

### change impact / stakeholder map例

```text
変更内容:
- 初回提案資料作成時に proposal skeleton、evidence matrix、review workflow board を使う
影響を受ける業務:
- 提案資料作成、FAQ作成、営業責任者レビュー、法務・情報システム確認
影響を受ける成果物:
- 提案資料、FAQ、根拠表、レビュー依頼文
影響を受ける人 / チーム:
- 営業担当、営業責任者、営業企画、法務、情報システム
変更される入力情報:
- 顧客質問の要旨、公開情報、社内標準説明
変更される出力形式:
- 提案骨子、FAQ、根拠表、未確認事項一覧
変更される承認フロー:
- 顧客共有前に営業責任者レビューを必須化し、価格・契約・セキュリティは専門部門確認へ回す
影響を受けるシステム / データ:
- 提案資料保管場所、レビュー管理ボード、社内AI環境
情報分類上の注意:
- 顧客名、価格、契約条件、内部 URL、個人名は入力しない
反対 / 不安が出やすい点:
- テンプレート入力の負荷、レビュー遅延、責任範囲の不明確さ
必要な教育 / FAQ:
- 入力禁止情報、レビュー状態の更新方法、エスカレーション条件
相談先:
- 営業責任者、法務、情報システム
合意が必要な相手:
- 営業責任者、営業企画マネージャー、情報システム責任者
```

### risk / dependency / assumption register例

| 項目 ID | 種別 | 内容 | 影響 | 対応 | 担当 | 状態 |
| --- | --- | --- | --- | --- | --- | --- |
| R-01 | リスク | 顧客固有情報をAIへ入力する | 情報漏えい、契約違反の可能性 | 入力禁止情報をFAQ化し、レビューで確認する | 営業企画 | 対応中 |
| D-01 | 依存関係 | セキュリティ標準文の最新版確認 | FAQ公開可否に影響 | 情報システムへ確認する | 情報システム | 確認待ち |
| A-01 | 前提 | 試行は社内AIのみで実施する | AI利用範囲の制約 | AI use boundary card に記載する | 営業企画 | 確認済み |
| U-01 | 未確認事項 | 価格表を別紙扱いにするか | 顧客共有範囲に影響 | 営業責任者と法務で判断する | 営業責任者 | 未確認 |

### 解説

変更影響を、関係者、成果物、データ、承認フローまで広げて確認しています。また、リスク、依存関係、前提、未確認事項を分けることで、何を管理すべきかが明確になります。

## 演習16-5：change request decision log と go / no-go checklist を作る

### change request decision log例

```text
変更要求ID: CR-001
要求内容: 顧客向けFAQにもAI生成ドラフトを利用したい
要求者: 営業担当
背景:
- 提案後の質問対応が増えており、回答案作成を標準化したい
影響する成果物:
- customer FAQ draft、evidence matrix、レビュー依頼
影響するスコープ:
- 社内レビュー資料から顧客向け資料へ対象が広がる
影響する期限:
- 次回顧客会議前までにレビューが必要
影響する品質 / リスク:
- 価格、契約、セキュリティ表現の未承認回答リスク
代替案:
- まず社内確認用FAQとして作成し、顧客共有版は承認後に作る
推奨案:
- 条件付き承認。顧客固有情報を入力せず、営業責任者と必要な専門部門のレビュー後に共有する
未確認事項:
- セキュリティ標準文の最新版、価格条件の扱い
承認者:
- 営業責任者、必要に応じて法務・情報システム
判断結果:
- 条件付き承認
判断理由:
- FAQの型は有用だが、対外共有には追加レビューが必要なため
次アクション:
- FAQテンプレートに根拠、未確認事項、承認者欄を追加する
ログ化先:
- change request decision log、review workflow board
```

### go / no-go checklist例

```text
対象: 初回提案資料AI支援の限定展開
判断日: レビュー会当日
判断者: 営業企画マネージャー、営業責任者
必須成果物がそろっているか:
- project outcome brief、KPI / leading indicator map、pilot evaluation plan、FAQ、相談先一覧がある
受け入れ基準を満たしているか:
- 提案資料ごとに目的、根拠、未確認事項、承認者が記録されている
未解決リスクは許容範囲か:
- 顧客固有情報の外部AI入力は行っていない。セキュリティ標準文は確認待ち。
情報分類とAI利用条件は確認済みか:
- 社内AI限定、入力禁止情報はFAQ化済み
承認者は明確か:
- 営業責任者。価格、契約、セキュリティは専門部門確認。
教育 / FAQ / 相談先は準備済みか:
- 初回説明資料、FAQ、相談先を用意済み
運用監視とログ化先は明確か:
- review workflow board と post-implementation learning report
ロールバック / 一時停止条件はあるか:
- 入力禁止情報の重大指摘が出た場合は一時停止
最終判断:
- conditional go
条件:
- 顧客向け正式回答には使わず、営業責任者レビュー後の資料に限定する
次アクション:
- 限定チームで展開し、次回レビュー会で評価する
```

### 解説

変更要求を記録し、影響、代替案、承認者を明確にしています。go / no-go では、未解決リスクがあるため、全展開ではなく conditional go としています。
