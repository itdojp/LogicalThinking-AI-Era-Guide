# 第15章 演習問題 解答・解説

この解答例は、唯一の正解ではありません。実務では、自社の情報分類、AI利用ルール、承認フロー、利用可能なツール、顧客契約、法務・セキュリティ・人事の所管部門判断に合わせて調整してください。

## 演習15-1：team operating brief を作る

### 題材: 営業提案チームの顧客向け資料ドラフト運用

```text
チーム名: 法人営業提案チーム
対象期間: 2026年5月-2026年8月
運用責任者: 営業企画マネージャー
対象業務: 顧客向け提案資料ドラフト、FAQ案、議事録要約、競合比較表、社内レビュー依頼
AI利用の目的: 初稿作成と論点整理の速度を上げ、レビュー観点を標準化する
期待する成果物: proposal skeleton、customer FAQ draft、meeting decision log、review workflow board
対象外業務: 価格条件、契約条件、正式回答、法的見解、セキュリティ例外、顧客固有情報を含む外部AI入力
主な読み手 / 利用者: 営業担当、営業責任者、法務、情報システム、顧客責任者
扱う情報分類: 社内限定、顧客情報、契約情報、価格情報、画面キャプチャ
利用できるAI環境: 社内AIのみ。顧客名、価格、個人名、契約条件は入力前に抽象化する
承認が必要な成果物: 顧客向け資料、FAQ、価格・契約に触れる回答、画面キャプチャ付き説明
レビュー担当: 一次レビューは営業責任者、専門レビューは法務・情報システム
最終承認者: 顧客責任者
ログ化先: review workflow board、prompt / template asset register、decision / escalation memo
再利用できる範囲: 顧客固有情報を除いた構成、確認観点、FAQの型
禁止事項: 外部AIへの顧客固有情報入力、未確認効果の断定、承認前の対外共有
未確定事項: セキュリティ表現の標準文、価格表共有可否の承認条件
次回見直し日: 月末レビュー会
```

### 解説

チーム単位の運用では、使うAI環境、対象外業務、承認が必要な成果物を先に定義します。便利な使い方だけでなく、止める条件とログ化先を明確にしている点が重要です。

## 演習15-2：AI use boundary card と responsibility / approval matrix を作る

### AI use boundary card例

```text
業務名: 顧客向けFAQドラフト作成
AIに任せること:
- 顧客質問の分類
- FAQドラフトの構成案
- 用語の説明案
- 未確認事項の洗い出し
AIに任せないこと:
- 契約条件、価格、提供可否、セキュリティ適合の最終判断
- 顧客への正式回答
- 法的見解の確定
入力してよい情報:
- 顧客名を伏せた質問の要旨
- 公開済み製品情報
- 社内で共有可能な標準説明
入力してはいけない情報:
- 顧客名、個人名、契約金額、未公開ロードマップ、内部URL、認証情報
匿名化 / 抽象化の条件:
- 顧客固有名詞を「顧客A」に置換し、価格・契約条件は入力しない
利用できる環境:
- 社内AIのみ
出力後に人間が確認すること:
- 根拠資料、契約条件への抵触、セキュリティ表現、未確認事項
承認が必要な場合:
- 顧客へ共有する場合、価格・契約・セキュリティに触れる場合
相談先:
- 法務、情報システム、営業責任者
ログ化する内容:
- 使用したテンプレートID、レビュー指摘、承認者、最終版
```

### responsibility / approval matrix例

| 成果物 | 作成者 | 一次レビュー | 専門レビュー | 承認者 | 最終責任者 | エスカレーション条件 |
| --- | --- | --- | --- | --- | --- | --- |
| FAQドラフト | 営業担当 | 営業責任者 | 情報システム | 顧客責任者 | 営業企画マネージャー | セキュリティ、契約、価格に触れる |
| 価格・契約回答 | 営業担当 | 営業責任者 | 法務、経理 | 権限者 | 営業責任者 | 標準条件から外れる |
| 画面キャプチャ説明 | 営業担当 | 営業責任者 | 情報システム | 顧客責任者 | 営業責任者 | 内部ID、個人名、内部URLがある |

### 解説

AIに任せる範囲と、人間が担う承認・責任を分けています。特に、顧客向け成果物では、ドラフト作成と正式回答の責任を混同しないことが重要です。

## 演習15-3：prompt / template asset register を登録する

### prompt / template asset register例

```text
資産ID: SALES-FAQ-001
名称: customer FAQ draft with evidence check
種別: プロンプト + 出力テンプレート
対象業務: 顧客向けFAQドラフト作成
利用目的: 顧客質問を分類し、回答案、根拠、未確認事項、承認要否を整理する
想定読み手: 営業担当、営業責任者、情報システム、法務
入力できる情報分類: 公開情報、社内で共有可能な標準説明、匿名化した顧客質問
入力禁止情報: 顧客名、個人情報、契約金額、未公開ロードマップ、認証情報、内部URL
前提条件: 顧客へ共有する前に営業責任者と必要な専門部門が確認する
プロンプト本文 / テンプレート本文:
「以下の顧客質問の要旨を、質問カテゴリ、回答案、根拠資料、未確認事項、承認要否に分けて整理してください。断定できないことは要確認にしてください。」
期待する出力形式:
- 質問カテゴリ
- 回答案
- 根拠資料
- 未確認事項
- リスク
- 承認者
- 顧客へそのまま送ってよいか
受け入れ基準:
- 事実、解釈、推奨、要確認が分かれている
- 根拠資料が示されている
- 価格、契約、セキュリティは要承認として明示されている
- 顧客固有情報が含まれていない
利用できるAI環境: 社内AIのみ
レビュー担当: 営業責任者
承認者: 顧客責任者
最終更新日: 2026-05-25
保守担当: 営業企画マネージャー
変更履歴: 初版登録
廃止条件: 製品説明、契約条件、セキュリティ標準文が変更された場合は見直し。保守担当不在なら利用停止。
```

### 解説

プロンプト本文だけでなく、入力禁止情報、受け入れ基準、承認者、廃止条件を登録しています。これにより、テンプレートを安全に再利用しやすくなります。

## 演習15-4：review workflow board と output quality gate rubric を設計する

### review workflow board例

```text
案件 / 成果物: 顧客A向けFAQドラフト
作成者: 営業担当
使用した資産ID: SALES-FAQ-001
情報分類: 顧客情報、契約情報を含む。社内AIのみ利用。
AI利用環境: 社内AI
現在の状態: 専門レビュー中
未解決論点:
- セキュリティ標準文の最新版確認
- 価格条件への言及を削除するか別紙扱いにするか
根拠確認の状態:
- 公開資料は確認済み
- 契約条件は法務確認待ち
リスク確認の状態:
- 顧客名は匿名化済み
- 画面キャプチャは未添付
承認者: 顧客責任者
期限: 次回顧客会議前営業日
次アクション:
- 情報システムがセキュリティ表現を確認
- 法務が契約条件への抵触を確認
ログ化先: review workflow board、decision / escalation memo
再利用可否: 顧客固有情報を除いたFAQ構成のみ再利用可
```

### output quality gate rubric例

```text
成果物名: 顧客向けFAQドラフト
読み手: 顧客担当者、顧客情報システム部門
利用場面: 次回会議後のフォロー資料
合格条件:
1. 質問ごとに結論、補足、根拠、要確認が分かれている
2. 価格、契約、セキュリティの断定が承認済みである
3. 出所、更新日、適用範囲が示されている
4. 顧客固有情報、個人情報、内部URLが不要に含まれていない
5. 未確認事項が「回答不可」ではなく「確認中」として整理されている
6. 顧客が次に何をすればよいかが分かる
7. 再利用できる構成と顧客固有部分が分かれている
差し戻し条件:
- 根拠がない断定がある
- 承認者不明の価格・契約表現がある
- 情報分類が未確認である
承認者: 顧客責任者
証跡: review workflow board、承認コメント、最終版FAQ
```

### 解説

レビューを「表現を直す場」ではなく、状態、根拠、リスク、承認、再利用可否を確認する工程として設計しています。差し戻し条件を明示すると、レビュー担当者ごとの判断ばらつきを減らせます。

## 演習15-5：knowledge reuse backlog と operating metrics review note を作る

### knowledge reuse backlog例

```text
項目ID: KR-20260525-01
種別: 改善
発見元: レビュー差し戻し
内容: FAQドラフトに根拠資料欄がなく、回答の出所確認に時間がかかった
対象資産ID: SALES-FAQ-001
期待効果: FAQレビュー時の根拠確認を短縮し、未確認の断定を減らす
リスク: 根拠欄が細かすぎると入力負荷が増える
担当者: 営業企画マネージャー
期限: 次回営業レビュー会まで
状態: 対応中
反映先: prompt / template asset register、output quality gate rubric
次回確認日: 月末レビュー会
```

### operating metrics review note例

```text
対象期間: 2026年5月
対象業務: 顧客向けFAQドラフト、提案資料レビュー
確認した成果物: FAQ 5件、提案資料 3件
主な利用資産: SALES-FAQ-001、proposal skeleton、risk review checklist
良かった点:
- FAQの構成がそろい、レビュー担当が確認しやすくなった
- 未確認事項が明示され、顧客への断定表現を減らせた
手戻りの原因:
- 根拠資料欄が任意だったため、出所確認で差し戻しが発生した
- セキュリティ標準文の最新版リンクが古かった
レビュー指摘の傾向:
- 根拠不足、承認者不明、未確認事項の書き方不統一
承認が遅れた理由:
- 法務確認が必要な条件を事前に判定できていなかった
リスクを検知できた例:
- 画面キャプチャに内部URLが残っていたため、共有前に差し止めた
再利用できた資産:
- FAQカテゴリ表、未確認事項欄、次アクション欄
廃止 / 更新が必要な資産:
- 古いセキュリティ標準文を含むテンプレート
次の改善アクション:
- 根拠資料欄を必須化する
- セキュリティ標準文を保守担当が月次確認する
- 法務確認が必要な条件をAI use boundary cardに追加する
担当者: 営業企画マネージャー
次回確認日: 2026年6月レビュー会
```

### 解説

利用回数ではなく、差し戻し理由、承認遅延、リスク検知、再利用された資産を振り返っています。指標は個人を評価するためではなく、テンプレート、レビュー観点、責任分界を改善するために使います。
