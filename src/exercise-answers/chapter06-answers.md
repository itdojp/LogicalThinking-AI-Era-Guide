# 第6章 演習問題 解答・解説

この解答例は、唯一の正解ではありません。実務では、利用目的、読み手、情報分類、承認者、社内ルールに合わせて調整してください。

## 演習6-1：CARE+でAI出力を評価する

対象のAI出力は、業務利用の初稿としては差し戻しが妥当です。

### CARE+評価

| 観点 | 評価 | 理由 | 次アクション |
| --- | --- | --- | --- |
| Correctness | 要修正 | 「ほとんどの企業」「恒常的に利用」の根拠がない | 公的統計、社内実績、調査定義を確認する |
| Appropriateness | 要修正 | 全社フルリモートへの即時移行は、社内事情や職種差を無視している | 職種、業務、顧客対応、セキュリティを分ける |
| Relevance | 要修正 | 投資判断、制度設計、移行条件の情報が不足している | 意思決定点を「移行可否」ではなく「試行条件」に絞る |
| Effectiveness | 要修正 | 実行手順、対象範囲、リスク対策がない | 試行部門、期間、KPI、判断会議を設定する |
| Evidence | 要修正 | 導入率、コスト削減、満足度向上の根拠がない | 出典、確認日、適用範囲を明記する |
| Risk | 要修正 | セキュリティ、労務、顧客対応、チーム運営のリスクが抜けている | risk review を追加する |
| Approval | 要修正 | 人事、情報システム、法務、部門長の承認条件がない | 承認者と確認項目を明記する |

### 採否判断

判断: 差し戻し。

理由: 未確認の一般化を根拠に、全社フルリモートへの即時移行を推奨しているためです。経営判断に使うには、根拠、職種差、移行条件、リスク、承認者が不足しています。

### 改善指示例

```text
改善指示:
テレワーク移行案を、全社即時移行ではなく、試行導入の判断メモとして作り直してください。

修正条件:
- 「ほとんどの企業」などの断定は削除し、確認済み情報と要確認情報を分ける。
- 社内の職種、業務、顧客対応、セキュリティ要件を分ける。
- 目的を、コスト削減だけでなく、生産性、採用、従業員体験、リスクの観点で整理する。
- 試行対象、期間、KPI、判断会議、承認者を明記する。
- 労務、情報セキュリティ、顧客対応、チーム運営のリスクと緩和策を表にする。
```

## 演習6-2：経営向け1枚サマリーのrubricを作る

### 必須条件

| 観点 | 採用可能 | 差し戻し |
| --- | --- | --- |
| 判断点 | 導入可否、試行可否、予算承認のどれを求めるか明確 | 何を決める資料か不明 |
| 費用 | 初期費用、運用費、契約期間、更新条件が分かる | 金額、期間、条件が不明 |
| 効果 | 業務課題と期待効果が対応している | 「効率化」など抽象表現だけ |
| 代替案 | 導入しない案、別サービス案、内製案がある | 推奨案だけで比較できない |
| リスク | セキュリティ、運用、契約、定着のリスクがある | リスクがない、または一般論だけ |
| 根拠 | 参照資料、確認日、社内ヒアリング結果がある | 根拠候補だけで確認済みではない |
| 承認条件 | 承認者、確認部門、残課題が明確 | 誰が何を承認するか不明 |

### 承認条件

- 情報システム部門がセキュリティ要件を確認している。
- 法務または購買担当が契約条件を確認している。
- 利用部門が試行目的とKPIに合意している。
- 経営会議で、予算、導入範囲、撤退条件を承認している。

### 差し戻し条件

- 導入効果が根拠なく断定されている。
- 費用や契約期間が不明である。
- 代替案や導入しない場合の影響がない。
- セキュリティや運用負荷のリスクが扱われていない。
- 承認者と残課題が明記されていない。

## 演習6-3：改善指示ログを作る

### 改善指示ログ例

| 項目 | 記録内容 |
| --- | --- |
| 対象成果物 | 新規サービス提案骨子 v1 |
| 用途 / 読み手 | 営業部長向けの提案方針確認 |
| 評価結果 | 根拠不足、リスク不足、担当者不明のため差し戻し |
| 問題点1 | 顧客課題と提案機能の対応が曖昧 |
| 修正指示1 | 顧客課題、根拠候補、提案機能、期待効果を表で対応させる |
| 問題点2 | 導入リスクが「運用に注意」としか書かれていない |
| 修正指示2 | 運用負荷、契約条件、既存システム連携、顧客教育の4観点でリスクと緩和策を書く |
| 問題点3 | 次アクションの担当者と期限がない |
| 修正指示3 | 営業、プリセールス、法務、顧客側担当者ごとに担当と期限を入れる |
| 再評価基準 | 根拠、リスク、担当、期限、承認条件がそろっていること |
| regression確認 | 初稿にあった顧客課題と提案骨子の対応関係を維持すること |

### 改善指示文例

```text
提案骨子v1を修正してください。

維持する点:
- 顧客課題を「問い合わせ対応の属人化」と整理している点。
- 提案の方向性を「FAQ更新と回答テンプレート整備」に絞っている点。

修正する点:
- 顧客課題、根拠候補、提案機能、期待効果を1つの表にしてください。
- 運用負荷、契約条件、既存システム連携、顧客教育のリスクを追加してください。
- 次アクションとして、担当者、期限、承認者、残課題を明記してください。
- 未確認情報は本文に入れず、要確認欄へ分けてください。
```

## 演習6-4：regression checklistを作る

### regression checklist例

```text
【顧客向け回答テンプレート regression checklist】
□ 回答対象の問い合わせ範囲が残っている
□ 根拠欄にFAQの版、確認日、参照箇所が残っている
□ 注意書き、前提条件、例外条件が消えていない
□ 「必ず」「すべて」「即時」などの保証表現が追加されていない
□ 顧客が次に行う手順、問い合わせ先、必要情報が残っている
□ 未確認の効果や数値が本文へ混ざっていない
□ サポート責任者と法務確認の承認条件が残っている
□ 顧客向けのトーンを保ちつつ、重要な制約が弱まっていない
□ 旧版で修正した誤表記や誤解を招く表現が復活していない
```

### 運用上の注意

短くする修正では、根拠や注意書きが消えやすくなります。文字数を削る場合も、根拠欄、要確認欄、承認条件は削除対象から外すように指定します。

## 演習6-5：採否判断メモを書く

### 採否判断メモ例

```text
【採否判断メモ】
対象成果物: 顧客向けFAQ回答テンプレート v3
用途: サポート担当者が一次回答に使う標準文面
情報分類: 公開可能情報。ただし社内運用手順は非公開。
判断: 条件付き採用

採用条件:
- FAQの版、確認日、参照箇所を根拠欄に残す。
- SLAを保証する表現を削除し、対応範囲と受付時間を明記する。
- サポート責任者が内容を確認する。
- 公開前に法務が過大表示の有無を確認する。

承認者:
- サポート責任者: 内容確認
- 法務担当: 表現確認
- CX責任者: 公開判断

確認日:
- FAQ確認: 2026-05-25
- サポート確認: 未了
- 法務確認: 未了

残課題:
- 例外対応の問い合わせ先を追加する。
- サポート時間外の案内文を確認する。

再利用条件:
- 同じサービスFAQの一次回答に限り再利用可。
- FAQ更新時、SLA変更時、法務指摘時は再評価する。
- 顧客別契約条件が関わる場合は個別確認する。
```

### 解説

条件付き採用では、「何を満たせば使えるか」を明確にします。承認者、確認日、残課題、再利用条件がない場合、後続担当者が採用済みと誤解する可能性があります。

## まとめ

- CARE+評価では、文章の自然さだけでなく、根拠、リスク、承認条件を確認します。
- 評価rubricは、採点表ではなく、差し戻し条件と承認条件を明確にする道具です。
- 改善指示ログには、問題点、修正指示、再評価基準、維持すべき条件を残します。
- regression checklist は、修正による品質低下を防ぐために使います。
- 採否判断メモには、判断理由、承認者、確認日、残課題、再利用条件を残します。
