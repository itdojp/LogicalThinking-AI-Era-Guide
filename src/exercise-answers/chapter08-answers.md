# 第8章 演習問題 解答・解説

この解答例は、唯一の正解ではありません。実務では、業務目的、情報分類、読み手、承認者、社内ルールに合わせて調整してください。

## 演習8-1：業務フロー組み込みマップを作る

### 例: 月次業績サマリー作成

```text
対象業務: 月次業績サマリー作成
成果物: 経営向け1枚サマリー
読み手 / 承認者: 事業部長、経営会議 / 経営企画責任者
意思決定点: 翌月の重点施策を変更するか
情報分類: 売上・顧客・粗利は社内限定。外部AIには投入しない。
```

### 工程別の分担

| 工程 | AIに任せる作業 | 人間が判断する作業 | ログ |
| --- | --- | --- | --- |
| 入口 | サマリーの構成案、比較軸案 | 読み手、意思決定点、利用データ範囲 | task brief |
| 作成 | 見出し案、説明文の初稿、要確認リスト | 使う数値、採用する比較軸 | drafting log |
| 評価 | CARE+観点の不足候補 | 採用、条件付き採用、差し戻し | CARE+評価シート |
| 検証 | 根拠候補の整理 | 社内正本、確認日、対象範囲 | evidence matrix |
| 承認 | 承認前チェックリストの初稿 | 最終版、残リスク、共有範囲 | decision log |
| 再利用 | 次月向けテンプレート候補 | 適用範囲、廃止条件 | template asset |

### analysis workflow logの併記例

月次業績サマリーのように分析を含む成果物では、業務フロー組み込みマップとは別に、analysis workflow log を残します。

```text
分析テーマ: 月次業績の変動要因整理
利用目的 / 読み手: 翌月重点施策を決める経営会議向け
意思決定点: 重点施策を維持するか、地域別・商品別に変更するか
データ分類: 社内限定。外部AIには実数値を渡さない
データ取得元: 社内BI、会計システム、営業管理システム
対象期間 / 母集団: 当月、前年同月、直近四半期。対象は確定済み売上のみ
集計単位: 月、地域、商品カテゴリ、チャネル
AIに渡した範囲: 匿名化した列名、レポート構成、分析観点
AIに渡していない範囲: 顧客名、案件名、契約金額、個人別成績
分析観点: 前年差、計画差、地域差、チャネル差
仮説: 新規案件減少、特定商品の失速、チャネル構成変化
確認済みの結果: 社内BIで確定した数値だけを記載
未確認 / 追加確認: 大口案件の時期ずれ、解約理由、キャンペーン影響
解釈上の注意: 相関を因果として断定しない
意思決定への示唆: 重点施策候補と要確認事項を分けて提示
承認者: 経営企画責任者
```

### 解説

重要なのは、AIに「月次業績を分析して」と丸投げしないことです。業績データは社内限定になりやすいため、外部AIを使う場合は列名や構成だけを扱い、実数値や顧客情報は社内環境または手作業で確認します。

## 演習8-2：research workflow sheetを作る

### research workflow sheet例

```text
調査テーマ: 新しい営業支援ツールを導入候補に入れるか
利用目的 / 読み手: 営業部長、情報システム、購買担当の一次判断
意思決定点: 詳細比較と試行に進むか
期限: 次回選定会議まで
情報分類: 公開情報は外部AI利用可。社内の商談データ、契約条件、顧客名は外部AI不可。
使ってよい情報: ベンダー公開Web、公開セキュリティ資料、公開導入事例
使ってはいけない情報: 既存顧客名、契約単価、営業個人の評価情報、未公開比較表
```

### source hierarchy例

| 階層 | 情報源 | 使い方 |
| --- | --- | --- |
| 一次情報 | ベンダー公式資料、価格ページ、セキュリティ資料 | 機能、価格条件、更新日確認 |
| 社内正本 / 社内確認済み情報 | 購買規程、情報システム基準、既存契約台帳 | 社内導入可否の条件確認 |
| 専門情報 | 業界団体資料、第三者レビュー | 論点候補として利用 |
| 報道 / 解説 | 導入事例記事、比較記事 | 一次情報へ遡る入口 |
| SNS / 口コミ | 利用者レビュー | 仮説候補のみ |
| AI出力 | 比較表の初稿、確認項目案 | 根拠にしない |

### evidence matrixへの反映

| 主張 | 情報源 | 確認状態 | 利用可否 |
| --- | --- | --- | --- |
| SSOに対応している | ベンダー公式資料 | 要確認 | 情シス確認後に利用可 |
| 初期費用が予算内 | 価格ページ、見積 | 未確認 | 本文断定不可 |
| 営業担当の入力負荷が下がる | AI出力、公開事例 | 要確認 | 試行で検証 |
| 既存CRMと連携可能 | 公式連携一覧 | 要確認 | 技術確認後に利用可 |

### 解説

AIは調査観点と比較表の初稿を作る補助に使います。導入可否の判断には、社内正本、ベンダー一次情報、情報システム確認、購買確認が必要です。

## 演習8-3：drafting / review loopを設計する

### 題材: 稟議メモ

```text
成果物: 新ツール試行導入の稟議メモ
読み手: 部門長、購買、情報システム
意思決定点: 3か月の小規模試行を承認するか
情報分類: ベンダー公開情報は利用可。見積、契約条件、社内利用部門名は社内限定。
```

### output schema例

```text
1. 申請目的
2. 背景 / 課題
3. 試行範囲
4. 選択肢
5. 推奨案
6. 費用の扱い（確認済み / 要確認）
7. 効果仮説
8. リスク / 緩和策
9. 承認条件
10. 次アクション
```

### acceptance criteria例

| 観点 | 受け入れ基準 | 差し戻し条件 |
| --- | --- | --- |
| Correctness | 費用、期間、対象範囲が確認済み | 未確認数値を断定している |
| Appropriateness | 稟議の読み手に必要な粒度 | 技術説明に偏りすぎている |
| Relevance | 試行承認の判断に直結 | 背景説明が多く判断点が弱い |
| Evidence | 根拠、確認日、確認者がある | AI出力だけを根拠にしている |
| Risk | 情報漏えい、運用負荷、契約リスクを記載 | リスクが抽象的 |
| Approval | 承認者と承認条件が明確 | 誰が承認するか不明 |

### 改善指示例

```text
稟議メモの初稿を修正してください。
- 冒頭に「3か月の小規模試行を承認するか」を明記する。
- 見積金額は確認済み / 要確認を分ける。
- 効果は仮説として書き、測定方法を追記する。
- 情報システム、購買、法務の確認条件を入れる。
- 承認後の次アクションを担当 / 期限付きで書く。
```

## 演習8-4：meeting / proposal workflow checklistを作る

### 例: 提案レビュー会議

#### 会議前

- 目的: 提案書を顧客提示可能な状態にするか判断する。
- 情報分類: 顧客名、契約条件、未公開価格を外部AIに入力しない。
- AIに依頼すること: 論点整理、想定反論、チェックリスト案。
- 人間が確認すること: 価格、契約条件、顧客事情、承認者。

#### 会議中

| 区分 | 記録すること |
| --- | --- |
| 決定事項 | 顧客に提示する主メッセージ、削除する記載 |
| 未決事項 | 価格条件、導入範囲、法務確認 |
| 要確認事項 | 顧客KPI、既存契約、セキュリティ条件 |
| ToDo | 担当、期限、成果物 |

#### 会議後

```text
【meeting / decision log】
決定事項:
- 提案の主軸は導入後支援に置く。
- 未確認の効果数値は本文から外し、試行後測定とする。
未決事項:
- 価格条件は購買確認後に反映する。
ToDo:
- 営業: 顧客KPIの確認
- PM: 導入スケジュール案の更新
- 法務: 契約条項の確認
共有範囲: 提案チーム内
承認者: 提案責任者
```

### 解説

会議でAIを使う場合、AIに合意事項を推測させないことが重要です。発言から判断できないものは、未決事項または要確認事項として残します。

## 演習8-5：multimodal intake cardを作る

### 題材: 公開PDFと社内スクリーンショット

```text
対象資料: 業界団体の公開PDF、社内ダッシュボードのスクリーンショット
形式: PDF / スクリーンショット
利用目的: 市場動向の調査メモと社内検討用サマリー
読み手: 企画部長、プロジェクトメンバー
情報分類:
- 公開PDF: 公開情報
- 社内スクリーンショット: 社内限定、外部AI投入不可
外部AI投入可否:
- 公開PDF: 利用可。ただし引用条件とページ番号を確認
- スクリーンショット: 利用不可。必要な情報は社内で匿名化した要約にする
匿名化 / マスキング要否: 顧客名、社内KPI、担当者名を除外
作成者 / 入手元: 業界団体Webサイト、社内BIツール
公開日 / 更新日 / 取得日: PDFの版と取得日を記録。スクリーンショットは取得日時を記録
加工履歴: 提案資料用に抜粋する場合は加工内容を記録
provenance確認: PDFの発行元、ページ、引用条件を確認
読み取ってよい範囲: PDFの公開テキスト、図表の見出し、ページ番号
読み取ってはいけない範囲: 社内スクリーンショットの数値、顧客名、担当者名
AIに依頼する作業: PDFの主張、数値、要確認事項の抽出
AIに判断させないこと: 自社への適用可否、投資判断、顧客向け利用可否
確認担当: 企画担当、法務または広報確認者
利用可否: 条件付き利用
```

### 解説

マルチモーダル資料は、AIが読み取れるかどうかより、扱ってよいかどうかが先です。公開PDFは利用できる場合でも、ページ、版、引用条件を確認します。社内スクリーンショットは、外部AIへ投入しない前提で、必要な情報を社内確認済みの要約に変換します。

## まとめ

- 業務フロー組み込みマップでは、AIに任せる作業、人間が判断する作業、承認が必要な作業を分けます。
- research workflow sheet は、AIを根拠ではなく調査設計と根拠候補整理に使うためのシートです。
- drafting / review loop は、初稿、評価、改善指示、再評価、承認を一連の品質管理にします。
- analysis workflow log は、データ定義、対象期間、母集団、解釈上の注意を残すために使います。
- meeting / proposal workflow checklist は、決定事項、未決事項、要確認事項、ToDoを分けるために使います。
- multimodal intake card は、PDF、画像、表、グラフ、スクリーンショットを扱う前に、情報分類、provenance、外部AI可否を確認するために使います。
